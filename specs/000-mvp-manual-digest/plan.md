## Summary

SignalFlow is a Claude Code skill/command system that generates high-quality digests from user-provided content. The MVP focuses on core value delivery: taking text snippets about a topic and synthesizing non-obvious insights, anti-patterns, and actionable experiments. The MVP deliberately avoids autonomous discovery and persistence to enable rapid iteration on the core insight-extraction engine. Post-MVP phases will add automation, learning, and source management.

## Technical Context

**Language/Version**: Claude Code Skills (native slash commands and prompts)
**Input**: Text snippets only (user pastes 3-5 snippets, 100-500 words each; NO URLs, NO web fetching)
**Processing**: Claude's text reasoning (pattern recognition, synthesis)
**Output**: Markdown files (date-stamped, minimal schema)
**Storage**: File system only (no database, no persistence in MVP)
**Testing**: Manual invocation + output validation
**Target Platform**: Claude Code IDE (skill system only)
**Project Type**: Single-command skill (no subagents, no MCP, no external code)
**Performance Goals**: Digest synthesis in < 3 minutes per run
**Constraints**: Claude Code execution model; token limits per request (3-5 snippets = ~2000-3000 tokens)
**Output Targets**: 1-3 insights, 2-4 anti-patterns, 1-3 actions, 3-5 resources (exact counts)

**Not in MVP**: MCP, APIs, web fetching, URLs, subagents, persistence, scheduling, feedback loops, learning

**Future Dependencies** (Phase 2+):
- Twitter API MCP (autonomous tweet discovery)
- WebSearch MCP (autonomous web search)
- WebFetch MCP (content extraction from URLs)
- Subagents (parallel content retrieval)

**Structure Decision**:

SignalFlow MVP is implemented as a **single Claude Code skill** (minimal scope):
- `/daily-digest <topic> "[text snippet 1]" "[text snippet 2]" "[text snippet 3]"` — text snippets only, no URLs, no web fetching
- Skill defined as `.md` file in `.claude/commands/daily-digest.md` with step-by-step prompts
- Generated digests archived in `digests/` by date
- Feature spec colocated in `specs/000-mvp-manual-digest/`
- **No data directory**: MVP has no persistent state; all input is ephemeral

**Rationale**:
- **Minimal scope**: Focus on core insight extraction first
- **No infrastructure overhead**: No persistence, no subagents, no MCP setup needed for Phase 1
- **Rapid iteration**: Edit prompts → test in 5 minutes → iterate
- **Fast delivery**: Get working system in 1-2 weeks, not months of infrastructure building
- **Validation**: Prove the core value (insight quality) works before automating discovery
- **Scalability path**: Phase 2 adds MCP (Twitter API, WebSearch, WebFetch) + subagents; Phase 3 adds persistence/learning

**Future Dependencies** (Phase 2+):
- Twitter API MCP (requires Twitter Developer credentials for autonomous tweet discovery)
- WebSearch MCP (for autonomous web content discovery)
- WebFetch MCP (for full-page content extraction)

## Complexity Tracking

**No violations at MVP stage**. The feature is intentionally scoped to single-user, file-based, no external databases or service dependencies. Future scaling (multi-user, cloud storage, scheduled execution) may introduce architectural changes, but MVP keeps it simple.

| Potential Future Complexity | Why Deferred | Simpler Current Approach |
|---------------------------|-------------|------------------------|
| Multi-user state/isolation | Not needed for single-user MVP | Single YAML/JSON files per user's project |
| Cloud storage (Notion, etc.) | Optional integration post-MVP | File-based local storage |
| Scheduled execution (cron) | Not needed for manual invocation MVP | User-triggered `/daily-digest` command |
| Real-time updates (WebSockets, SSE) | Not needed for daily batch digest | Polling/batch mode sufficient |
| Feedback capture & learning | Deferred to Phase 3+ | Manual digest generation only |
