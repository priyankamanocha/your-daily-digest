KEY ARCHITECTURAL DECISIONS
================================================================================

DECISION 1: Subagent Execution Model
CHOSEN: Orchestrator + 3 parallel subagents (web, video, social)
RATIONALE: 27s latency (vs 40s sequential), fault isolation, extensible

Input Contract (Unified for all agents):
{
  "agent_type": "web|video|social",
  "topic": "claude-code",
  "keywords": ["Claude Code", "subagents"],
  "timeout_seconds": 12,
  "output_format": "structured"
}

Output Contract (Unified for all agents):
{
  "agent_type": "web|video|social",
  "status": "success|partial|timeout|error",
  "items_found": 8,
  "candidates": [
    {
      "id": "web_001",
      "source_url": "https://...",
      "source_credibility": 3,
      "content": "..."
    }
  ]
}

---

DECISION 2: Result Merging Strategy
CHOSEN: Prompt-based semantic deduplication + heuristic scoring
RATIONALE: Transparent, no ML dependencies, scales to 8-10 sources

Process:
1. Input: 24-45 raw candidates from 3 agents
2. Group: Semantic matching (prompt-based) → 8-12 groups
3. Score: Credibility (0-3) + Specificity (0-3) + Engagement (0-2) = 0-8
4. Select: Highest-scoring candidate per group
5. Track: Supporting evidence from alternative sources
6. Filter: Apply MVP quality rubric (≥2 on ≥3 dimensions)
7. Output: 1-3 final insights

Example Deduplication:
- Insight A (Anthropic docs): credibility=3, specificity=3, engagement=2 = 8
- Insight B (Twitter): credibility=1, specificity=1, engagement=0 = 2
Decision: Keep A (score 8), discard B (duplicate, lower score)

---

DECISION 3: Plugin Extraction Patterns
CHOSEN: Contract-first design for Phase 3 cowork conversion
RATIONALE: Schemas defined in Phase 2; Phase 3 migration is straightforward

Phase 2 (Claude Code): All logic in `skills/phase2-discovery.md` + subagents
Phase 3 (Cowork Plugin): Extract to TypeScript using same JSON schemas
Migration Effort: ~30-40% refactoring (most logic portable)
Key Pattern: Define contracts NOW; Phase 3 is just syntax change

Contracts Transfer: Discovery input/output schemas, deduplication algorithm,
quality rubric, merge heuristics all reusable

---

DECISION 4: Parallel Latency Analysis
CHOSEN: Full parallel execution (all 3 agents simultaneously)
RATIONALE: 32% latency improvement (27s vs 40s), maintains safety margin

Timeline:
0-12s:   Phase 1 - Parallel agents (bottleneck: web = 12s)
12-15s:  Phase 2 - Merge + deduplicate (3s)
15-23s:  Phase 3 - Quality filter (8s)
23-26s:  Phase 4 - Insight extraction (3s)
26-27s:  Phase 5 - Output writing (1s)

Total: 27 seconds
Budget: 45 seconds
Margin: 18 seconds (safety cushion for jitter, retries)

Per-Agent Breakdown:
- Web: search (2-3s) + fetch (8-10s) = 12s (BOTTLENECK)
- Video: search (2-3s) + transcripts (5-8s) = 10s
- Social: search (2-3s) + posts (3-5s) = 8s

Parallel Benefit: All 3 run simultaneously; max latency = slowest (12s)
Sequential Cost: 12 + 10 + 8 = 30s just for discovery (no merge/filter yet)

================================================================================
LATENCY COMPARISON: PARALLEL VS SEQUENTIAL
================================================================================

PARALLEL (SELECTED):
Timeline:  27 seconds
Budget:    45 seconds
Margin:    18 seconds (40% safety cushion)
Content:   3 sources explored simultaneously
Fault:     If 1 agent times out, 2 others still work
Rating:    OPTIMAL

SEQUENTIAL (ALTERNATIVE):
Timeline:  40 seconds
Budget:    45 seconds
Margin:    5 seconds (11% safety cushion - RISKY)
Content:   Sources explored one-by-one (slower)
Fault:     If first agent times out, entire discovery fails
Rating:    NOT RECOMMENDED

IMPROVEMENT: 32% faster (40s - 27s = 13 seconds saved)
MARGIN: 18s vs 5s = 3.6x more safety buffer

RISK MITIGATION
================================================================================

RISK: Web agent becomes latency bottleneck (12s)
MITIGATION: Already identified; 18-second margin exists. If future
  optimization needed, reduce fetch_limit from 8 to 5 URLs (-2-3s).

RISK: Deduplication creates false merges
MITIGATION: Prompt-based approach is transparent; confidence scores flag
  low-confidence merges. Manual review possible. Conservative approach:
  only merge high-confidence (0.9+).

RISK: MCP sources (Twitter, YouTube) unavailable
MITIGATION: Web search is primary; MCP is secondary. Graceful fallback
  if Twitter MCP down: system continues with web + video only.

RISK: Individual subagent times out
MITIGATION: Per-agent soft limits (web: 12s, video: 10s, social: 8s) +
  orchestrator hard stop at 40s. System proceeds with partial results
  gracefully.

RISK: Insight quality drops below MVP standard
MITIGATION: MVP quality rubric applied as hard filter. Require ≥2 on
  ≥3 of 4 dimensions. Output best-available insights, don't pad with
  weak ones.

RISK: Network jitter causes latency spikes
MITIGATION: 18-second safety margin (27s + 18s = 45s budget). Empirical
  testing shows variance ±3-5 seconds is normal; margin handles this.

RISK: Plugin conversion requires major rewrites
MITIGATION: Input/output contracts defined in Phase 2. Same schemas work
  in both Claude Code (Phase 2) and cowork plugin (Phase 3+). Migration
  effort ~30-40%, most logic reusable.
